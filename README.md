# Agile Planning Practice: Counter Service Backlog

A hands-on Agile/Scrum planning exercise: building, labeling, and prioritizing a product backlog for a sample "Counter" web service using GitHub as the planning tool.

Completed as part of the IBM Introduction to Agile Development and Scrum course and kept public as a working example of how I turn product requirements into a sprint-ready backlog.

Focus: product backlog management and sprint planning. This repo is about planning the work, not application code.

### What this repo demonstrates

Translating product requirements into user stories with clear acceptance criteria
Structuring and prioritizing a product backlog in GitHub Issues
Triaging work with labels (enhancement, technical debt, bug)
Standardizing intake with issue templates (.github/ISSUE_TEMPLATE)
Organizing a sprint backlog / kanban board in GitHub Projects (To Do → In Progress → Done)
Keeping new feature requests and technical debt visible side by side

### The product: A Counter service

The sample product is a simple counter web service. The backlog captures the features and technical work needed to take it from basic functionality toward a deployable product:

Backlog items & Type:                
Must allow multiple counters: User Story
Counter can be reset: Enhancement
Ability to update a counter to a new value: Enhancement
Ability to remove a counter: User Story
Complete deploy service to the cloud: Technical debt

Each item is framed from the user's perspective and prioritized for sprint planning.
